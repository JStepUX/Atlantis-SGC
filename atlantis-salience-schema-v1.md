# Atlantis Salience Schema v1.0

**Purpose:** Define the YAML frontmatter schema emitted by the Atlantis processing pipeline for every chunk written to a salience database. This schema is domain-agnostic — it must describe BJJ techniques, Minnesota trivia, Star Wars lore, couscous recipes, and anything else with equal fidelity.

**Consumers:** SGC orchestrator (retrieval at runtime), Chroma vector DB (metadata filtering), humans (debugging and inspection).

**Design principle:** The structure is the aperture. The content is the light that passes through it. Domain lives in the *values*, never in the *field names*.

---

## Full Schema

```yaml
---
# =============================================================================
# IDENTITY
# =============================================================================
# Unique addressing. Every chunk is locatable by chunk_id alone.
# document_slug groups chunks from the same source.
# source_file is the raw input path for provenance tracing.

chunk_id: "bjj-kimura-grip-mechanics_003"
document_slug: "bjj-kimura-grip-mechanics"
source_file: "Data/raw/combat/kimura-grip-mechanics.md"

# =============================================================================
# CONTENT DESCRIPTION
# =============================================================================
# What is this chunk, what does it say, and what conceptual space does it occupy?

chunk_type: "procedure"
# Enum: section | definition | procedure | fact | idiom | narrative | reference
#
# section     — a structural division of a larger document
# definition  — explains what something is
# procedure   — explains how to do something (steps, mechanics, technique)
# fact        — a discrete, verifiable claim
# idiom       — a figure of speech, saying, or cultural expression
# narrative   — a story, account, or descriptive passage
# reference   — a lookup table, list, catalog, or index

document_title: "Kimura Grip Mechanics from Closed Guard"

summary: "Breaking grip resistance when opponent answers the phone during kimura attempt"
# 1-2 sentences. What would you tell someone who asked "what's in this chunk?"
# Generated by the small model during Atlantis ingest.

topics:
  - topic: "kimura"
    depth: 0
  - topic: "grip-mechanics"
    depth: 0
  - topic: "closed-guard"
    depth: 1
  - topic: "bjj"
    depth: 2
  - topic: "grappling"
    depth: 3
  - topic: "martial-arts"
    depth: 4
# Depth scale:
#   0 = this chunk is ABOUT this topic (the subject)
#   1 = immediate technical context (the frame)
#   2 = the discipline or domain it belongs to
#   3 = the broader field
#   4 = the widest meaningful category
#
# Not every chunk uses all depths. A standalone fact might have depth 0 and 2 only.
# The small model assigns topics and depths during ingest.
# topic_path is derived deterministically by sorting topics by depth ascending
# and joining with dots: "kimura.grip-mechanics.closed-guard.bjj.grappling.martial-arts"

topic_path: "kimura.grip-mechanics.closed-guard.bjj.grappling.martial-arts"
# Computed field. Enables prefix-match hierarchical retrieval.
# "Give me everything under bjj.*" = all BJJ chunks regardless of technique.

aliases:
  bjj: "Brazilian Jiu-Jitsu"
  closed-guard: "full guard"
  kimura: "double wristlock"
# Same-concept, different-name mappings. Max 3-4 entries.
# Concatenated into embedding input during Atlantis processing.
# Rule: if you have to think about whether it's an alias, it's not.

# =============================================================================
# NAVIGATION
# =============================================================================
# Chunk-to-chunk traversal within a single source document.

navigation:
  current: 3
  total: 7
  next: "bjj-kimura-grip-mechanics_004"
  previous: "bjj-kimura-grip-mechanics_002"

series: "bjj-fundamentals"
# Optional. Groups documents that form a sequence or collection.

related_chunks:
  - "bjj-americana-from-mount_002"
  - "bjj-wrist-control-principles_001"
# Chunks from OTHER documents that are conceptually related.
# Populated by Atlantis using topic overlap or cross-reference detection.

# =============================================================================
# INDEX REFERENCE
# =============================================================================
# Back-reference to the materialized index built during Atlantis processing.
# Enables fan-out: chunk -> index entry -> siblings -> their chunks.

index_ref:
  entry: "brazilian-jiu-jitsu"
  line: 132
# Points to the categorical index.md entry for this chunk's root grouping.
# If the chunk doesn't belong to a group (tcr < threshold), this is null.

# =============================================================================
# SALIENCE
# =============================================================================
# Numeric signals the orchestrator uses to rank and filter retrieval results.
# All scores are 0.0-1.0 floats computed at ingest time.

salience:
  lexical_distinctiveness: 0.71
  # TF-IDF divergence from the corpus at ingest time.
  # High = this chunk uses rare/distinctive terminology.
  # Low = this chunk uses common vocabulary shared by many chunks.
  # Computed during the TF-IDF pre-vectoring pass.

  information_density: 0.85
  # How much meaning per token. A dense technical paragraph scores high.
  # A padded narrative introduction scores low.
  # Approximated by: (unique non-stopword terms / total tokens).

  specificity: 0.90
  # How narrow/concrete vs. broad/abstract.
  # "The kimura requires figure-four grip control" = high specificity.
  # "Grappling involves controlling your opponent" = low specificity.
  # Estimated by depth-0 topic density relative to total topics.

  standalone: true
  # Can this chunk be understood without its neighbors?
  # false = contains references like "as described above" or depends on prior context.
  # The orchestrator uses this to decide whether to pull adjacent chunks.

# =============================================================================
# GOAL BINDING
# =============================================================================
# How does this chunk relate to what an agent might be DOING?

goal_affinity:
  - domain: "combat-sports"
    weight: 0.95
  - domain: "body-mechanics"
    weight: 0.60
# Domain-weight pairs. Domain is a free-text label, not an enum.
# Weight is the strength of relevance (0.0-1.0).
# The orchestrator filters by domain when it knows the agent's current goal.
# Max 4 entries per chunk. If everything is relevant, nothing is.

utility: "procedural"
# Enum: procedural | declarative | contextual | orienting
#
# procedural  — tells you HOW to do something
# declarative — tells you WHAT something is
# contextual  — tells you WHY something matters right now
# orienting   — tells you WHERE you are in a bigger picture
#
# An agent pursuing a goal needs different utility types at different stages:
#   orienting early, procedural mid-task, contextual when surprised.

# =============================================================================
# PROVENANCE
# =============================================================================
# Where did this information come from and how much should you trust it?

provenance:
  source_type: "human-authored"
  # Enum: human-authored | model-generated | system-event | conversation
  #
  # human-authored  — written by a person (document, article, book)
  # model-generated — produced by an AI model (summary, synthesis)
  # system-event    — emitted by a system (log, webhook, ticket update)
  # conversation    — extracted from a dialogue

  authority: "canonical"
  # Enum: canonical | derived | informal | speculative
  #
  # canonical   — authoritative source (textbook, official doc, primary source)
  # derived     — synthesized from authoritative sources (notes, summaries)
  # informal    — casual or unverified (forum post, offhand remark, hearsay)
  # speculative — theory, hypothesis, untested idea
  #
  # Orthogonal to source_type: a human can write something speculative,
  # a model can generate something canonical (if summarizing a textbook).

  confidence: 0.95
  # Epistemic reliability. 1.0 = verified fact. 0.5 = plausible but unconfirmed.
  # A peer-reviewed paper gets 0.95. A reddit comment gets 0.3.

  author: null
  # Attribution if known. Null if anonymous or not applicable.

  original_source: null
  # Upstream URL, ISBN, document reference if applicable.

# =============================================================================
# LIFECYCLE
# =============================================================================

status: "active"
# Enum: active | draft | superseded | archived
#
# active      — current and valid
# draft       — work in progress, not yet authoritative
# superseded  — replaced by a newer chunk (see related_chunks for successor)
# archived    — retained for history but no longer operationally relevant
#
# The orchestrator can exclude non-active chunks from retrieval
# or weight them lower.

# =============================================================================
# TEMPORAL
# =============================================================================

temporal:
  created_at: "2026-06-21T14:00:00Z"
  # When Atlantis processed and created this chunk.

  effective_date: null
  # When the information in this chunk became true/valid.
  # Null = no specific effective date (evergreen or unknown).

  stale_after: null
  # When this chunk should be considered potentially outdated.
  # Null = no expiry. The orchestrator can flag or deprioritize stale chunks.

  decay_class: "evergreen"
  # Enum: evergreen | slow-decay | fast-decay | ephemeral
  #
  # evergreen  — always valid (mathematical fact, stable technique, idiom)
  # slow-decay — valid for months/years, eventually outdated (product docs, policies)
  # fast-decay — valid for days/weeks (sprint status, current events)
  # ephemeral  — valid for hours (live system state, in-progress conversation)
  #
  # The orchestrator maps these to retrieval behavior:
  #   evergreen = always eligible
  #   ephemeral = only if created_at is within a recency window

  last_accessed: null
  # Updated by the orchestrator at retrieval time. Not set during ingest.

  access_count: 0
  # Updated by the orchestrator at retrieval time. Not set during ingest.
  # These two fields let the system learn which chunks are actually useful.

# =============================================================================
# CONTENT
# =============================================================================

tokens: 312
# Estimated token count of the body content (below the frontmatter).
# Used by the orchestrator for context window budget management.

---

When the opponent answers the phone to defend the kimura, the critical
mechanic is driving your elbow toward their far hip rather than trying
to muscle the arm backward...
```

## Field Summary

| Section | Fields | Set by | Used by |
|---|---|---|---|
| Identity | chunk_id, document_slug, source_file | Atlantis (deterministic) | Orchestrator, human debugging |
| Content Description | chunk_type, document_title, summary, topics, topic_path, aliases | Atlantis + small model | Embedding model, orchestrator, Chroma filtering |
| Navigation | navigation, series, related_chunks | Atlantis (deterministic) | Orchestrator fan-out |
| Index Reference | index_ref | Atlantis (post-TF-IDF) | Orchestrator fan-out to sibling chunks |
| Salience | lexical_distinctiveness, information_density, specificity, standalone | Atlantis (TF-IDF + heuristics) | Orchestrator ranking |
| Goal Binding | goal_affinity, utility | Small model during ingest | Orchestrator filtering by agent goal |
| Provenance | source_type, authority, confidence, author, original_source | Small model + config | Orchestrator trust weighting |
| Lifecycle | status | Manual or pipeline rules | Orchestrator inclusion/exclusion |
| Temporal | created_at, effective_date, stale_after, decay_class, last_accessed, access_count | Atlantis (ingest) + orchestrator (runtime) | Orchestrator recency gating |
| Content | tokens | Atlantis (computed) | Orchestrator context window budgeting |

## Index File Specification

The categorical index is a materialized file (`index.md` or `index.json`) built by Atlantis after processing a corpus. It is NOT part of the chunk frontmatter — it lives alongside the Chroma DB as a companion artifact.

### Construction Logic

1. During ingest, identify **root terms** per source document: proper nouns (or high-TF-IDF terms) that meet a repetition threshold relative to document length.
2. If a source document has >= N distinct roots (threshold TBD empirically), create an index entry for it.
3. After all documents are processed, run TF-IDF cosine similarity across all index entries to detect **siblings** (entries with similarity above threshold).
4. Write the index with entries, their children (chunk references), and their siblings (other entries with high TF-IDF similarity).

  "entries": {

### Retrieval Fan-Out Pattern

1. Query matches chunk `bjj-kimura-grip-mechanics_003`
2. Chunk's `index_ref` points to entry `"brazilian-jiu-jitsu"` 
3. Entry's `children` list provides sibling chunks in the same domain
4. Entry's `siblings` map provides related entries (e.g., `"body-mechanics"`)
5. Orchestrator can optionally pull chunks from sibling entries if context budget allows

## Open Questions

1. **Root detection threshold** — what repetition count (absolute or relative) qualifies a term as a root? Needs empirical testing against real corpora.
2. **Sibling similarity threshold** — what TF-IDF cosine similarity score constitutes "sibling"? Needs calibration by inspecting the similarity matrix on a test corpus.
3. **Small model selection** — which model handles the classification tasks (chunk_type, summary, topics, aliases, goal_affinity, utility, authority, confidence, standalone)? Candidates: local model on 5090, Haiku via API, or a fine-tuned small model.
4. **Alias concatenation strategy** — append aliases to body text before embedding, or prepend as a header block? Affects embedding geometry.
5. **topic_path ordering** — currently depth-ascending (specific to general). Confirm this is the right order for prefix matching use cases.
6. **Chroma collection strategy** — one collection per DB, or one collection per index entry? Single collection is simpler; per-entry enables faster scoped queries.
7. **Access tracking persistence** — last_accessed and access_count are updated at runtime. Where does the orchestrator write these back? Directly to Chroma metadata, or to a separate lightweight store?
